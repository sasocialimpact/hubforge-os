    96→
    97→### Key design decisions
    98→
    99→1. **No socket.io** - The frontend orchestrates the reasoning loop via sequential HTTP API calls. This works on Vercel's serverless platform (no persistent connections needed).
   100→
   101→2. **API keys in browser** - User API keys (OpenAI, Groq, etc.) are stored in localStorage and sent directly to the provider. They never touch HubForge servers.
   102→
   103→3. **localStorage-first** - All user data (programs, org profile, context blocks, usage) is stored in localStorage. Supabase is optional for persistence across devices.
   104→
   105→4. **Serverless-friendly** - Each API route is a standalone function with `maxDuration` set. No state between requests.

   210→## 3. The 9-Engine Reasoning Pipeline
   211→
   212→The core of HubForge OS is a recursive reasoning loop with 9 engines. The frontend orchestrates the loop by calling API routes sequentially.
   213→
   214→### Pipeline flow
   215→
   216→```
   217→User input (problem text)
   218→    │
   219→    ▼
   220→┌──────────────────┐
   221→│ 1. SUPERVISOR    │  POST /api/interview
   222→│ Decomposes       │  → objectives, scope, stakeholders
   223→│ problem          │  → clarifying questions
   224→│ Asks questions   │  LLM call (Z.ai/OpenAI/Groq/etc.)
   225→└────────┬─────────┘
   226→         │ decomposition
   227→         ▼
   228→┌──────────────────┐
   229→│ 2. RETRIEVAL     │  POST /api/run-step { step: 'retrieval' }
   230→│ Pulls from       │  → frameworks, decision rules, evidence
   231→│ Knowledge Graph  │  → historical memory, reasoning patterns
   232→│ (no LLM)         │  Deterministic (no API cost)
   233→└────────┬─────────┘
   234→         │ retrieval result
   235→         ▼
   236→┌──────────────────┐
   237→│ 3. WEB SEARCH    │  POST /api/search
   238→│ Searches live    │  → demographic data
   239→│ web for context  │  → previous programs in area
   240→│ (3 parallel)     │  → evidence/research
   241→│                  │  3 web searches + LLM summary
   242→└────────┬─────────┘
   243→         │ web search results
   244→         ▼
   245→┌──────────────────┐
   246→│ 4. RULE ENGINE   │  POST /api/run-step { step: 'rule' }
   247→│ Deterministic    │  → SMART goal check
   248→│ validation       │  → stakeholder coverage
   249→│ (no LLM)         │  → assumption/risk/evidence checks
   250→└────────┬─────────┘
   251→         │ rule check results
   252→         ▼
   253→    ┌────────────────────────────────────┐
   254→    │        RECURSIVE LOOP              │
   255→    │  (repeats up to MAX_ITERATIONS)    │
   256→    │                                    │
   257→    │  ┌──────────────────┐              │
   258→    │  │ 5. REASONING     │  LLM call    │
   259→    │  │ Drafts strategy  │  (most       │
   260→    │  │ using all context│   expensive) │
   261→    │  └────────┬─────────┘              │
   262→    │           │ draft                  │
   263→    │           ▼                        │
   264→    │  ┌──────────────────┐              │
   265→    │  │ 6. CRITIQUE      │  LLM call    │
   266→    │  │ Finds weaknesses │  Checks      │
   267→    │  │ (7 heuristics)   │  assumptions,│
   268→    │  └────────┬─────────┘  evidence,   │
   269→    │           │ critique   vagueness   │
   270→    │           ▼                        │
   271→    │  ┌──────────────────┐              │
   272→    │  │ 7. IMPROVEMENT   │  LLM call    │
   273→    │  │ Rewrites to fix  │  Fixes every │
   274→    │  │ every issue      │  critique    │
   275→    │  └────────┬─────────┘  issue       │
   276→    │           │ improved draft          │
   277→    │           ▼                        │
   278→    │  ┌──────────────────┐              │
   279→    │  │ 8. EVALUATION    │  LLM call    │
   280→    │  │ Scores 0-100 on  │  6 criteria: │
   281→    │  │ 6-criterion       │  Evidence,   │
   282→    │  │ rubric            │  SMART,      │
   283→    │  └────────┬─────────┘  Feasibility, │
   284→    │           │            Stakeholders,│
   285→    │           │            Causal Logic,│
   286→    │           │            Risks       │
   287→    │           ▼                        │
   288→    │    score >= threshold?             │
   289→    │    YES → exit loop                 │
   290→    │    NO → loop again                 │
   291→    └────────────────────────────────────┘
   292→         │ final draft + score
   293→         ▼
   294→┌──────────────────┐
   295→│ 9. STRUCTURE     │  POST /api/structure
   296→│ Extracts ToC     │  → Theory of Change JSON
   297→│ + Logframe       │  → Logframe JSON
   298→│ from strategy    │  LLM call
   299→└────────┬─────────┘
   300→         │ structured outputs
   301→         ▼
   302→    Deliverable view
   303→    (strategy + ToC + Logframe)
   304→    → Editable
   305→    → Exportable (Word/PDF/Excel)
   306→    → Feedback loop
   307→```
   308→
   309→### Engine implementations
   310→
   311→All engines are in `src/lib/engines.ts`. Each engine:
   312→
   313→1. Takes typed inputs
   314→2. Constructs a system prompt + user prompt
   315→3. Calls the LLM via the `llm()` function (with timeout + retry)
   316→4. Parses the response (JSON extraction for structured outputs)
   317→5. Returns typed outputs
   318→
   319→```typescript
   320→// Example: Reasoning Engine signature
   321→export async function reasoningEngine(
   322→  config: ProviderConfig,           // AI provider config
   323→  problem: string,                  // User's problem text
   324→  decomposition: Decomposition,     // From Supervisor
   325→  retrieval: RetrievalResult,       // From Retrieval Engine
   326→  priorCritique: string | null,     // From previous iteration's Critique
   327→  priorDraft: string | null,        // From previous iteration's draft
   328→  pack: DomainPack,                 // Knowledge pack
   329→  iteration: number,                // Current iteration (1-5)
   330→  maxIterations: number,            // Max iterations (configurable)
   331→  outputTypes: OutputType[],        // What outputs to produce
   332→  answers?: Record<string, string>, // User's interview answers
   333→  webSearch?: WebSearchResult,      // Web search results
   334→  orgContext?: string,              // Organization context
   335→  contextBlocks?: string,           // Reusable context blocks
   336→): Promise<string>                  // Returns the draft (Markdown)
   337→```
   338→
   339→### LLM Router
   340→
   341→The `llm()` function routes to the correct provider:
   342→
   343→```typescript
   344→// Z.ai (default, uses SDK)
   345→if (config.provider === 'zai') {
   346→  const zai = await getZAI()
   347→  const completion = await zai.chat.completions.create({ ... })
   348→  return completion.choices[0]?.message?.content
   349→}
   350→
   351→// Z.ai with user's own key (OpenAI-compatible endpoint)
   352→if (config.provider === 'zai-key') {
   353→  // Falls through to OpenAI-compatible fetch
   354→}
   355→
   356→// All other providers (OpenAI, Anthropic, Gemini, Groq, Local)
   357→const res = await fetch(`${baseUrl}/chat/completions`, {
   358→  method: 'POST',
   359→  headers: { 'Authorization': `Bearer ${apiKey}`, ... },
   360→  body: JSON.stringify({ model, messages, temperature }),
   361→})
   362→```
   363→
   364→### Timeout + Retry
   365→
   366→Every LLM call has:
   367→- **90-second timeout** (configurable in Geek Mode)
   368→- **2 retries** with exponential backoff (3s, 6s)
   369→- **Empty response check** (throws if LLM returns empty)
   370→
   371→```typescript
   372→const LLM_TIMEOUT_MS = 90000
   373→const LLM_MAX_RETRIES = 2
   374→
   375→for (let attempt = 1; attempt <= LLM_MAX_RETRIES + 1; attempt++) {
   376→  try {
   377→    // Call LLM with timeout
   378→    const result = await withTimeout(llmCall, LLM_TIMEOUT_MS)
   379→    if (!result.trim()) throw new Error('Empty response')
   380→    return result
   381→  } catch (e) {
   382→    if (attempt <= LLM_MAX_RETRIES) {
   383→      await new Promise(r => setTimeout(r, attempt * 3000))
   384→    }
   385→  }
   386→}
   387→```

   391→## 4. Knowledge Graph (8 Layers)
   392→
   393→The Social Impact Pack knowledge graph is defined in `src/lib/knowledge.ts`. It has 8 layers that feed different engines:
   394→
   395→| Layer | Name | Contents | Used by |
   396→|-------|------|----------|---------|
   397→| 1 | Domain Knowledge | Sectors (Education, Health, WASH, etc.) | Supervisor |
   398→| 2 | Framework Knowledge | 6 frameworks (ToC, Logframe, Outcome Mapping, MSC, Impact Eval, Survey Design) | Retrieval Engine |
   399→| 3 | Procedural Knowledge | Program design process, evaluation process | Supervisor |
   400→| 4 | Decision Rules | 5 deterministic rules (SMART, Stakeholders, Assumptions, Evidence, Risk) | Rule Engine |
   401→| 5 | Evidence Libraries | 5 evidence sources (OECD-DAC, Better Evaluation, Innosight, World Bank, CGAP) | Reasoning Engine |
   402→| 6 | Historical Memory | 3 past cases (farmer livelihoods, girls' education, rural healthcare) | Reasoning Engine |
   403→| 7 | Reasoning Patterns | 6 patterns (Root Cause, Counterfactual, Tradeoff, Risk, Comparative, Contribution) | Reasoning Engine |
   404→| 8 | Improvement Heuristics | 6 heuristics (Weak assumptions, Missing evidence, Vague→measurable, Causal logic, Inconsistency, Reduce uncertainty) | Critique Engine |
   405→
   406→### Adding to the knowledge graph
   407→
   408→Edit `src/lib/knowledge.ts`:
   409→
   410→```typescript
   411→export const socialImpactPack: DomainPack = {
   412→  // ... existing fields ...
   413→  frameworks: [
   414→    // Add a new framework
   415→    {
   416→      name: 'My New Framework',
   417→      layer: 2,
   418→      description: 'What it does',
   419→      whenToUse: 'When to use it',
   420→      keyElements: ['element1', 'element2'],
   421→      template: 'IF... THEN... PROVIDED THAT...',
   422→    },
   423→  ],
   424→  decisionRules: [
   425→    // Add a new rule
   426→    {
   427→      name: 'My Rule',
   428→      layer: 4,
   429→      check: 'What to check',
   430→      passCondition: 'When it passes',
   431→      failAction: 'What to do if it fails',
   432→    },
   433→  ],
   434→  // ... etc for evidence, historicalMemory, reasoningPatterns, improvementHeuristics
   435→}
   436→```

   440→## 5. Domain Packs
   441→
   442→A Domain Pack is a self-contained knowledge module. The Social Impact Pack is the first. To create a new pack:
   443→
   444→1. Create a new file: `src/lib/health-pack.ts`
   445→2. Define the pack with the same `DomainPack` interface
   446→3. Import it in the API routes you want to use it with
   447→4. Add a pack selector in the UI (optional)
   448→
   449→```typescript
   450→// src/lib/health-pack.ts

   734→## 8. Libraries Reference
   735→
   736→### engines.ts
   737→The heart of the system. Contains:
   738→- `ProviderId` type: 'zai' | 'zai-key' | 'openai' | 'anthropic' | 'gemini' | 'groq' | 'local'
   739→- `ProviderConfig` interface
   740→- `normalizeConfig()`: fills defaults for each provider
   741→- `llm()`: the single LLM entry point with timeout + retry
   742→- `extractJSON()`: robust JSON extraction from LLM responses
   743→- `supervisorEngine()`: decompose problem + ask questions
   744→- `retrievalEngine()`: pull from knowledge graph (deterministic, no LLM)
   745→- `ruleEngine()`: validate against decision rules (deterministic, no LLM)
   746→- `reasoningEngine()`: generate draft strategy (LLM)
   747→- `critiqueEngine()`: find weaknesses (LLM)
   748→- `improvementEngine()`: fix weaknesses (LLM)
   749→- `evaluationEngine()`: score on rubric (LLM)
   750→- `structureEngine()`: extract ToC + Logframe (LLM)
   751→- `feedbackEngine()`: incorporate user feedback (LLM)
   752→
   753→### providers.ts
   754→7 AI provider definitions:
   755→- Z.ai (shared, free) - QUICK START badge
   756→- Z.ai (own key) - RECOMMENDED badge
   757→- Groq (free tier) - FREE TIER badge
   758→- OpenAI - pay per use
   759→- Anthropic (Claude) - pay per use
   760→- Google Gemini - free tier
   761→- Local (Ollama) - FREE FOREVER badge

   770→

   772→- `Program` interface (id, title, status, problem, draft, evaluation, structured, etc.)

   785→- `exportStrategyToWord()`: Markdown → .docx (uses `docx` library)

   791→### api-client.ts
   792→Frontend API client. Every call is tracked for usage:
   793→```typescript
   794→async function apiCall(path: string, body: any): Promise<any> {
   795→  const start = Date.now()
   796→  const res = await fetch(path, { method: 'POST', ... })
   797→  const json = await res.json()
   798→  // Track usage
   799→  trackUsage({
   800→    provider: body.providerConfig?.provider || 'zai',
   801→    engine: body.step || path.split('/').pop(),
   802→    durationMs: Date.now() - start,
   803→    inputChars: JSON.stringify(body).length,
   804→    outputChars: JSON.stringify(json).length,
   805→  })
   806→  return json
   807→}
   808→```
   809→

   811→- `trackUsage()`: record an API call (provider, engine, duration, chars)
   812→- `getUsageSummary()`: aggregate by provider, engine, 7-day chart
   813→- `getOptimizationTips()`: suggestions to reduce cost
   814→- `clearUsage()`: reset

   816→### analytics.ts
   817→Non-blocking event tracker:
   818→- `track()`: queue event, flush every 2 seconds

   824→## 9. AI Provider System
   825→
   826→### Provider IDs
   827→```typescript
   828→type ProviderId = 'zai' | 'zai-key' | 'openai' | 'anthropic' | 'gemini' | 'groq' | 'local'
   829→```
   830→
   831→### Provider Config
   832→```typescript
   833→interface ProviderConfig {
   834→  provider: ProviderId
   835→  apiKey?: string    // stored in browser localStorage only
   836→  baseUrl?: string   // OpenAI-compatible endpoint
   837→  model?: string     // model name
   838→}
   839→```
   840→
   841→### How keys work
   842→
   843→1. User selects provider in Settings dialog
   844→2. If provider needs key, API key field appears
   845→3. Key stored in `localStorage` under `hubforge.providerConfig`
   846→4. Key sent directly to provider with each API call (via fetch)
   847→5. Key NEVER sent to HubForge servers
   848→
   849→### Adding a new provider
   850→
   851→1. Add to `ProviderId` type in `providers.ts` and `engines.ts`
   852→2. Add to `DEFAULT_BASE_URLS` and `DEFAULT_MODELS` in `engines.ts`
   853→3. Add to `PROVIDER_LABELS` in `engines.ts`
   854→4. Add to `PROVIDERS` array in `providers.ts` with metadata
   855→5. If the provider is OpenAI-compatible, it works automatically via the `fetch()` path in `llm()`
   856→6. If not OpenAI-compatible, add custom routing in `llm()`

   860→## 10. Data Flow: End-to-End
   861→
   862→```
   863→1. User types problem in General Mode
   864→   → stored in `problem` state
   865→
   866→2. User clicks "Help me build it"
   867→   → POST /api/interview (Supervisor Engine)
   868→   → Returns decomposition + clarifying questions
   869→
   870→3. User answers questions (or skips)
   871→   → Answers stored in `answers` state
   872→
   873→4. User clicks "Build my strategy + ToC"
   874→   → runLoop() starts
   875→
   876→5. runLoop() calls sequentially:
   877→   a. POST /api/run-step { step: 'retrieval' }
   878→      → Returns frameworks, rules, evidence from knowledge graph
   879→   
   880→   b. POST /api/search
   881→      → Returns demographics, previous programs, evidence from web
   882→      → Auto-saved as context block
   883→   
   884→   c. For each iteration (1 to maxIterations):
   885→      i.   POST /api/run-step { step: 'rule' }
   886→      ii.  POST /api/run-step { step: 'reasoning' }
   887→           → Includes: orgContext, webSearch, retrieval, answers
   888→      iii. POST /api/run-step { step: 'critique' }
   889→      iv.  POST /api/run-step { step: 'improvement' }
   890→      v.   POST /api/run-step { step: 'evaluation' }
   891→      vi.  If score >= threshold: break
   892→   
   893→   d. POST /api/structure
   894→      → Returns ToC JSON + Logframe JSON
   895→
   896→6. Deliverable shown to user
   897→   → Strategy (Markdown, rendered)
   898→   → ToC (visual diagram, editable)
   899→   → Logframe (table, editable)
   900→
   901→7. Auto-save:
   902→   → Program saved to localStorage (programs.ts)
   903→   → Session saved to /api/memory (if Supabase configured)
   904→   → Web search saved as context block (context-blocks.ts)
   905→
   906→8. User can:
   907→   → Edit ToC/Logframe inline
   908→   → Export to Word/PDF/Excel
   909→   → Give feedback → POST /api/feedback → revised draft
   910→
   911→9. Every API call tracked:
   912→   → usage-tracker.ts records provider, engine, duration, tokens
   913→   → Usage panel shows consumption + optimization tips
   914→
   915→10. Every user action tracked:
   916→    → analytics.ts records events (app_open, run_start, etc.)
   917→    → Admin dashboard shows funnel, scores, errors
   918→```

  1081→## 16. Security Model
  1082→
  1083→### API Key Storage
  1084→- User API keys stored in browser `localStorage` only
  1085→- Keys sent directly to AI provider (OpenAI, Groq, etc.) via `fetch()`
  1086→- Keys NEVER sent to HubForge OS servers
  1087→- Keys NEVER logged to console
  1088→
  1089→### Input Validation
  1090→All API routes validate inputs:
  1091→- `problem`: required, string, max 10000 chars
  1092→- `feedback`: required, max 5000 chars
  1093→- `eventType`: alphanumeric + underscore only, max 100 chars
  1094→- All analytics fields: sanitized with `.slice()` length limits
  1095→- `user_agent` and `referrer`: truncated to 500 chars
  1096→
  1097→### Admin Protection
  1098→- Admin endpoints require `admin_key` query parameter
  1099→- Default key: `hubforge-admin-2024` (change via `HUBFORGE_ADMIN_KEY` env var)
  1100→- Wrong key returns 403
  1101→
  1102→### Rate Limiting
  1103→- Not implemented (Vercel handles per-IP limits)
  1104→- `maxDuration` on all routes prevents infinite execution
  1105→
  1106→### Data Encryption
  1107→- HTTPS/TLS 1.3 in transit (Vercel default)
  1108→- Supabase: AES-256 at rest
  1109→- localStorage: not encrypted (browser security model)

  1127→## 18. How to Extend
  1128→
  1129→### Add a new Domain Pack
  1130→
  1131→1. Create `src/lib/health-pack.ts` with `DomainPack` interface
  1132→2. Import in `src/lib/engine-access.ts`
  1133→3. Add pack selector in the UI (or hardcode for now)
  1134→4. Update API routes to use the new pack
  1135→
  1136→### Add a new engine
  1137→
  1138→1. Implement the engine function in `src/lib/engines.ts`
  1139→2. Add a new `step` case in `src/app/api/run-step/route.ts`
  1140→3. Add the call in `src/lib/api-client.ts`
  1141→4. Call it in the frontend loop (`general-mode.tsx` or `geek-mode.tsx`)
  1142→
  1143→### Add a new output type
  1144→
  1145→1. Add to `OutputType` in `src/lib/types.ts`
  1146→2. Add to `OUTPUT_OPTIONS` in `src/lib/types.ts`
  1147→3. Add handling in `reasoningEngine()` (include in prompt)
  1148→4. Add handling in `structureEngine()` (extract from draft)
  1149→5. Add a renderer component for the output
  1150→
  1151→### Add a new AI provider
  1152→
  1153→1. Add to `ProviderId` type in `providers.ts` and `engines.ts`
  1154→2. Add to `DEFAULT_BASE_URLS`, `DEFAULT_MODELS`, `PROVIDER_LABELS` in `engines.ts`
  1155→3. Add to `PROVIDERS` array in `providers.ts`
  1156→4. If OpenAI-compatible: works automatically. If not: add custom routing in `llm()`
  1157→
  1158→### Add a new export format
  1159→
  1160→1. Implement the export function in `src/lib/export-utils.ts`
  1161→2. Add a button in the deliverable view (`general-mode.tsx`)
  1162→
  1163→### Add a new page
  1164→
  1165→1. Create `src/app/my-page/page.tsx`
  1166→2. Add `'use client'` if it has interactivity
  1167→3. Add link in footer or navigation
  1168→
  1169→### Add a new API route
  1170→
  1171→1. Create `src/app/api/my-route/route.ts`
  1172→2. Export `async function POST(req: NextRequest)` (and/or GET)
  1173→3. Add `export const maxDuration = 60`
  1174→4. Add input validation
  1175→5. Wrap in try/catch
  1176→6. Add to `api-client.ts` if called from frontend
